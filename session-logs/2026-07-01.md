## Session 2 — Policy gate: no deployment without logged + approved mission

- **When:** 2026-07-01 (continued, ~12:15 UTC-4)
- **Context:** MSO directive: "We need to enforce trips, vehicles should not be deployed without the mission getting logged and approved. We can't work around this just to make timecards expedient, the team needs to do what is necessary to document the deployment." This inverts the earlier auto-create-trip idea — the answer is not to let drivers skip the trip-logging step, but to block departure when the step was skipped. Confirmed policy specifics via structured Q&A:
  - **Gate location:** both DVC submit AND Start trip.
  - **Approval source:** mission-level (`missions.approval_status = 'approved'`); the denormalized `trips.approval_status` field is ignored for gating.
  - **Auto-link from `efac24b`:** removed. Driver must explicitly pick the trip in the DVC form.

### What was already in place

- `missions` table with a real approval workflow (`approval_status ∈ {draft, pending, approved, rejected, revision_requested}`, `approved_by_id`, `approved_at`, `rejection_reason`), exposed via `/api/missions/[id]` actions `submit` / `approve` / `reject` / `request-revision`.
- `trips.mission_id REFERENCES missions(id)` and `trips.vehicle_id` — the trip is the execution row linked to the approved intention.
- `evaluateReadinessForMissionLinkedTrip` in `src/lib/mission-deployment-readiness.ts` already enforces `mission.approval_status = 'approved'` + `lifecycle_status = 'active'` + `assigned_vehicle_id` matches, and is called from `/api/trips/[id]/depart`. So the "Start trip" side of the gate was mostly built — what was missing was making the mission_approval / mission_lifecycle / mission (not-found) gates **non-overridable**, and the entire DVC-submit side.

### Changes (commit `23d0e3c`)

1. **DVC form (`src/components/DriverVehicleCheckForm.tsx`)** — Added a "Mission / Trip" picker that renders for departing direction only. Fetches approved-mission trips for the selected vehicle from the new `/api/trips/eligible-for-departure` endpoint. Required — submit is blocked with a clear message when no trip is selected or when the eligible list is empty ("No approved mission for this vehicle today. Ask dispatch to log and approve a mission before you depart."). The previously-selected trip is cleared if it falls out of the eligible list (e.g. another session started it, or the mission was un-approved). Returning direction is ungated.

2. **DVC POST (`src/app/api/driver-vehicle-checks/route.ts`)** — Server-side hard gate for `direction = departing`:
   - Requires `tripId`.
   - Validates the trip exists; `vehicle_id` matches the DVC's vehicle; `mission_id` is set; linked mission `approval_status = 'approved'`; linked mission `lifecycle_status = 'active'`; `departed_at IS NULL`; `checkin_at IS NULL`.
   - Returns 400 with a distinct `reason` on each failure path (`missing_trip`, `trip_not_found`, `vehicle_mismatch`, `missing_mission`, `mission_not_approved`, `mission_not_active`, `trip_already_departed`, `trip_already_checked_in`) so the driver / dispatch gets an actionable message.
   - **Removed** the auto-link-to-active-trip logic from `efac24b`. Under the new policy the driver must explicitly pick the trip — silent linking undermined the approval gate (and was the original failure mode that left HR blind to the 13/06 LS SEH deployment).

3. **Start trip (`src/app/api/trips/[id]/depart/route.ts`)** — Tightened the override path. The mission-level gates (`mission_approval`, `mission` not-found, `mission_lifecycle`) are now **non-overridable**: an approver with `canOverridePrerequisite` permission can still bypass operational gates (DVC freshness, mechanical) with a reason, but cannot bypass a non-approved mission — they have to approve the mission first. Returns 403 with `reason: "mission_not_approved_no_override"` (or `mission_not_found_no_override` / `mission_${lifecycle}_no_override`) on the blocked paths.

4. **New endpoint (`src/app/api/trips/eligible-for-departure/route.ts`)** — `GET /api/trips/eligible-for-departure?vehicleId=X&org=Y` returns trips where `vehicle_id` matches, `mission_id` is set, linked mission `approval_status = 'approved'` and `lifecycle_status = 'active'`, `checkin_at IS NULL`, `departed_at IS NULL`, ordered by `planned_departure_date DESC` then `checkout_at DESC`, limit 50. Source of truth for the DVC picker; also reusable for a future "Start trip eligibility" badge on the Trips page.

### #1 — Audit endpoint (commit `5f5037d`)

New: `GET /api/admin/deployment-audit?org=1pwr_lesotho` — fleet-management roles only. Returns four lists plus a `counts` summary:

- `orphanTrips`: trips with `mission_id IS NULL`
- `tripsLinkedToUnapproved`: trips whose mission `approval_status != 'approved'` (or mission row missing)
- `orphanDepartingDvcs`: departing DVCs with `trip_id IS NULL`
- `dvcsLinkedToOrphanTrip`: DVCs whose `trip_id` points to a trip with `mission_id IS NULL` (or trip row missing)

Each list is capped at 200 rows, ordered by `created_at` / `checkout_at` DESC. The endpoint is read-only and intended for one-time / on-demand hygiene checks — dispatch can call it via curl or a future admin page can render it.

### #3 — Tighten `MISSION_PROFILE.LOCAL` to require DVC (commit `5f5037d`)
