## Session — HR-facing deployments API missing LS SEH field deployments

- **When:** 2026-07-01 (UTC-4 / New York)
- **Context:** HR portal flagged Rafube (1PWR63) and Motiki Ramothule (1PWR72) at site SEH (Sehlabathebe, Lesotho) as "OT Pre-Approval Required" for Saturday 27/06, blocking their timecard submissions. They have been on site since the 13/06 P1 deployment. HR's `user_ls_field_deployment_window` was returning NULL because `GET /api/deployments/current?employee_id=…` on fm.1pwrafrica.com was returning 404 for both employees — meaning the Fleet side never surfaced the deployment to HR. This session covers the full scope: data diagnosis + backfill on production, three product fixes to prevent recurrence, and verification on the HR side.

### Diagnosis on production (af-south-1, i-0dda937da2c9d0018, /var/www/fleet-hub/data/fleet-hub.db)

Ran the diagnostic SQL against the production SQLite via EC2 Instance Connect. Findings:

- The 13/06 P1 departing DVC exists (`dbb6bbb7-9b12-4ba1-9d1b-f41226e9b3f4`, route `HQ (Maseru) → Sehlabathebe`, driver Pemane Rafube, mileage 218852, overall_pass=1).
- `passenger_manifest` was `[]` (no passengers selected at submit time). HR's `/api/deployments/current` joins on `passenger_manifest[*].employee_id`, so 0 matches → 404 for both employees.
- `trip_id` was NULL (the DVC was never linked to a trip). Even with a manifest, `/api/deployments/current` derives `status="active"` from `trips.departed_at` via `dvc.trip_id`, so a NULL trip_id would still produce a non-active or pending_departure record.
- No trip row existed for P1 around 13/06 (the only P1 trip in 2026 was 2026-02-17 to SEB). No trip to Sehlabathebe/SEH existed at all in 2026.

Rafube and Ramothule can now submit their week-of-22-06 timecards (the 27/06 Saturday falls within the active window).

### Product fixes (commit `efac24b`)

Three changes to prevent the same root cause from recurring:

- **`src/components/DriverVehicleCheckForm.tsx`** — Client-side validation in `handleSubmit`: reject submit when `direction === "departing"` and `passengerManifest.length === 0`, with a message explaining the HR dependency (the manifest is the canonical record of who is on site and the anchor for HR's 6-week deployment clock). Returning checks remain ungated since they legitimately have no passengers.
- **`src/app/api/driver-vehicle-checks/route.ts`** — After the DVC INSERT, if no explicit `tripId` was supplied by the caller, auto-link the DVC to the unique still-active trip (`trips.vehicle_id` matches, `date(trips.checkout_at) = dvc.check_date`, `trips.checkin_at IS NULL`). Only links when exactly one candidate exists to avoid silently binding to the wrong trip. This closes the gap where a DVC could be created with passengers but no trip link, leaving `/api/deployments/current` unable to derive `status="active"`.
- **`src/app/api/driver-vehicle-checks/[id]/route.ts`** — Added `normalizePassengerManifest` (mirrors POST) and PATCH handling for `passengerManifest`, so HR/admins can correct manifests on existing checks without re-submitting the whole form. Empty arrays are allowed (returning checks); entries without an `employee_id` are dropped.
